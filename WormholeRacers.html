Wormhole Racers Design Document.

Wormhole Racers is a racing game where the race course is a three dimensional vortex that responds to the Racing vehicles inside of it.

The Raceway is a tube with an irregular geometry designed to mimic the inside of a vortex. The tube forms itself between two points in three dimensional space, and will shake, writhe and twirl during the race, with the individual racers affecting the vortex in different ways. Ripples on the walls (floor and ceiling) of the vortex will impede and accelerate the racers for example, and can provide ramps and surfaces for jumps and leaps.Gravity within the vortex operates outwards away from the centre, and this makes flying exceptionally difficult, so instead the ships roll and ride on energy shields that can be rotated and deformed to allow the ship to steer slide and jump (by compressing the shield “vertically”, sliding for a while and then expanding the shield to bounce up away from the vortex wall).

Each ship should be individually composed of components that are tuned to different frequencies, these frequencies are transmitted from the ship to the larger vortex, with each ship’s individual tunings all interfering in complex ways to contort, and twist the path, additionally the position of the ship within the vortex, both rotationally around the wall and position along the wormhole should also alter and affect the interactions, to make them more difficult to learn. Ships that are further ahead should be more affected by the positions of ships further back, and those near the front should be smoothing and creating easier paths behind them to allow other ships to access shorter, faster paths than were available to the leaders. Ship components can be salvaged, constructed or purchased throughout the Galaxy the game is set in, with several powerful interstellar companies all competing, with each favouring slightly different “scales” of tuning that are not entirely compatible, but are sometimes surprisingly compatible depending upon the exact nature of your ships construction.  

The game is based in a Universe where faster than light travel across the Galaxy is only possible through what are called Wormholes, these are Hyperspatial structures that are generated between Subspace Vortex Generators. Different Vortex Generators have different harmonic frequencies and sometimes connecting two generators will create chaotic and turbulent vortices that are more active to race along. The Galaxy will be procedurally generated with a number of species, including humans. These are presented as 2d animations, rather than 3d, this should allow individual portraits to be constructed from sprite sheets for each species along with a palette.

The game is based on a ship with multiple player character roles that should ideally be a multiplayer experience, but could be reduced to a single player on the ship with NPCs filling the other roles. The crew consists of Pilot (actually pilots the race parts), Navigator (often also the pilot when a Player who sets the destinations and usually decides cargo and so on), Engineer (responsible for fitting upgrades, ship’s wiring tuning (the length of the wires and where they are bent create subharmonics that effect the ships systems and the vortex, they also have to run around the inside of the ship and fix things that break during a race, and may have to retuned engine components on the fly (usually by having predesigned scales they can shift between), and Weapons operators who if they are played are generally just gunners with no control over the ship, although weapons fire will have an impact on the vortex because everything should.

The core game loop should appear as follows.

  1. Race other ships along a Vortex to a new system, the exact nature of the race will depend on the character and story mode, for example the story of a rookie pilot in a backwater league would be very different to a racer in the Galactic League. I feel Leagues should probably unlock by completing various “Stories” that are effectively tutorials and skill gates, to stop rookie players from trying to race in advanced games.
  2. Having arrived in the system the character will interact with others in the space docks and various procedurally generated point of interest in the system such as “League Offices”, “Scrapyards”, “Shops”, “Restaurants”, “Clubs”, “Bars”, “Factories” and so on. Different plots will seed the procedural generation with different options, for example a criminal story about smuggling and illegal races would add Criminal characters who operate in these fields who may make offers to the characters, that the character will have to consider the risks and rewards of both accepting and rejecting these offers. Just as a “Sponsor” plot may offer high spec components in exchange for control of the racers image. Character such as other racers will be commonly met who may have friendly co-racer vibes, competitor vibes, rival vibes or even villain vibes. We will need a fairly robust but flexible system for modelling these “friendliness” levels (that should consist of four or five different vectors, from honour, trust, love, financial concerns, and aggression (that may not be enough axes and details as I feel each “Story” should probably add at least one axis of its own.
  3. By performing jobs, or spending money in the system, or by selling trade goods legally, or a variety of other methods, the ship’s crew manage to gather upgrades or repairs for their ship. This exposes the engineering screen where the engineer can modify the ship, rebuilding the whole thing if they have enough components, but usually just making repairs or small upgrades.
  4. Sometimes particular systems will have orbital racing leagues in the system (this is probably where a rookie would begin scraping together cash from small jobs to enter the orbital league in their system). Orbitals are a circular race where the planetary bodies in the system are the main source of distortion on what would otherwise be an elliptical (close to circular) loop. Although the vortex drives of the racers will add wobbles and ripples to the structure, the gravity of the planets in the system will often create loops around those bodies, although such loops will never exactly intersect the vortex, they may come quite close to touching their outerwalls, and a carefully controlled accident may be able to push through as a shortcut, but will probably collapse the loop behind itself as fast as the loops empties of active drives. Smaller and less interesting systems will not have a scheduled race though. And this will force the racers to leave for another system. Rather like the game Elite the racers will often weigh their ship down with additional cargo that might turn a profit, and find some other ship willing to head the same way so that they can open a wormhole. If they can’t find another small ship willing to perform a run, they will have to sneak alongside a larger vessel that has multiple synced Vortex drives inside it (such as a liner, or freighter, or even a warship) that is headed in the right direction and try not to get crushed between the large vessel and the wormhole walls. Once past the ship they will have an easy ride to the ships destination, but no say in  where they are actually going. (This is probably a tutorial mission the first time the rookie has to fly to another system to make a race)
  5. The ship and crew may pass through a few systems until they make it to their next big planned race. And there are events that can happen randomly (each based on a story or plot line - such as the Pirate story that adds pirate raids, and the war story that adds Warships and space battles. Each story should effectively be treated as a DLC that can be turned on or off, or potentially developed in turn. The simplest random events will probably be “racing challenges” where some specific racer the player is familiar with from an earlier race will challenge the rookie to a new specific race, such as specific routes that are considered complicated because the wormhole passes past a number of stellar objects that affect the wormhole geometry knotting and twisting it in complex way beyond the normal.

Ships should be represented by a 3d model that has an interior that can be explored and altered by the engineer, and explored by the gunners and pilot if they specifically leave their seats (they may swap positions in game or assist the engineer with repairs for example). The Pilot should see from within the cockpit, and has controls that could fly the ship (and will if using the thrusters and pointing the nose up, but such flights are usually short and can involve crashes into the wall of the vortex that are so hard that the ship exist the wormhole in a random system along the route. Ideally ships should be purchaseable, but also rebuildable from scrap, with a whole host of different companies producing basic and advanced chassis that are more or less compatible with other peoples systems. Ships will require, a generator system for power, flight systems, vortex generators, shield generators, weapons systems, control computers, navigation computers, life-support systems, potentially repair bays and engineering machinery for tool crafting and so on.

There are a number of key points that need to be noted continually during development.

  1. All systems must be constructed in a modular manner. Every system should be isolated so that we can alter the components that construct it.
  2. Systems will probably require separate prototypes and their own construction phases. For example Any alterations to the way a ship is constructed will have implications for handling of the ship. And impact the tuning of the ship (which in turn will impact the stability of the wormhole).
  3. In order to test the ships we will probably require a space flight system as well as the wormhole system, we may as well also add a “Ground Race” where the ships flight has been disabled and it can only roll on its shield. This will also be a useful testing ground for the driving mechanics of the shield and the interaction with the engineering system, possibly it would make an excellent tutorial “episode” in the Rookie Storymode.
  4. I suspect the main menu will require a lot of iterations as we add test and remove game modes. For example I have realised that a “Hyperspace” race could have a floor above and below, but not have the wormhole wobble, allowing ships to fly, as long as they can avoid the hills and inverted hills, or they can roll on those surfaces and leap between. This also adds another potential set of league races, although I feel as though Hyperspace engines may have impacts on wormhole systems, so being fast at both would be very tricky. Again there are going to be balance issues galore.
  5. I think the game at minimum should have a few built in stories, these are not necessarily exposed to the player as choices, I suspect all new games should start the same way, with a beat up, old ship somewhere on a random planet (we will probably gate spiral arms as we add stories, with at least one of the spiral sarms being “at war” probably due to a inter-galactic invasion out on that arm. The invaders will have a completely different FTL system (which could be the Hyperspace system I guess or Hyperspace could be used by large ships, I suspect the aliens use something akin to warp bubbles, in that they travel through “real” space by moving the space itself). The ship is functional, low on fuel, and some distance from the space port. I think that early game choices can lead to rapid advances for the player if they wish (such as leaving their original crew behind to permanently join a professional racing crew after their own (probably pilot) is killed in a crash.
  6. This implies a structure of complex interacting systems, which I’m not sure how to develop with AI assistance for rapid prototyping. It may be worth generating the system as a series of different javascript files that are built to communicate with each other through a local database. I’m not sure even what the best system to build this game is, for some reason I suspect it will be most flexible if it is built in html and javascript although that is only because I think a certain amount of peer to peer communication would be easily offloaded that way, where as the final production model will have to be constructed in Unreal Engine for maximum performance. (That said if the js prototype is effective then good enough).
  7. Core to the system I think the first system has to be the loader. That will allow us to load the other systems, the loader will have to display a suitable UI. I think each module should have text or an image it displays as each module loads. So that when all the modules are loaded the main menu is revealed in full with each section that began invisible finally fully revealed and displaying whatever its “image” text or model is. That will be important, as modules are loaded various menus and UI elements will appear or unlock.
  8. It strikes me that during play we will require the players to share calculations (with at least 3 randomly chosen players performing essential game calculations on the tunnel, and sharing the results to the others. Any players inside the ship and not in a turret or the pilot seat are effectively in a completely different space to the race. So they will not need as much information about the shape of the wormhole or nature of hyperspace etc
  9. I’m not sure how to handle crew mates meeting inside the ship. I think they should see a physical model of them in a generic space suit (a few shapes can cover a lot of potential creatures), but there should be a 2d cartoon image of them when they talk. Unless we can get a truly cell shaded character mode without a helmet. I don’t mind it not being star wars to keep the workload down.

Game modes and Leagues

  • Planetary Leagues: Every inhabited planet has a planetary League. These are typically a combination of ground races and orbitals wormholes that just orbit around the planet, and form a large elliptical loop that is distorted by the other bodies in the system. Other planets, the moons and the sun will “tug” the perfect circle out of shape in a variety of ways. The ships are very small, limited in power and very limited in range. They have very cheap components and typically are at most able to the have a crew of 5 (although most have 1-3).
  • Local System League: These are larger ships generally (1-8 crew), more expensive , and can orbit the star. These races typically race “around” the planets, creating huge distortions in the wormhole route, which can twist around larger planets almost completely (accounting for moons) but can never intersect itself.
  • Local Interstellar League: This depends on where in the galaxy you are as different areas of the galaxy have different leagues that all have their own unique rules. But in general, these are larger ships that are more powerful and more capable, often with built in robot crew that can operate as backups or allow a solo pilot to collect all the winnings… These ships also have a Molecular scoop, which allows them to scoop matter from planetary atmospheres and bodies that the wormhole passes through. This can be used as a weapon in times of war, depleting enemy resources, but we’ll get to that later.This system basically means we can also mine resources by tunnelling through a planet, which will be essential to the engineer, and so the engineer will want to get scoops as soon as they can. Scoops can also be used to scoop up fuel and components from crashes (and perhaps survivors).
  • Rim Leagues: Very long races out on the outer rim, considered “wild” there are very few restrictions in place out here, and most of these leagues are at least partially criminal in nature, with weapons fire and armour being encouraged by the few rules in place.
  • Arm / Spur Leagues: The rules of races in each arm are usually set along the arm. Individual spurs are considered separate leagues, but they often use very similar rules to the main arm they are spur of. These are very tightly controlled races, but they allow limited and restricted ship builds and weaponry. Armour is a requirement in most of the arm leagues, but a few spurs are considered “safer” and do not mandate armour for every ship.
  • Core League: The Galactic core has a single league that covers all races. No weaponry is allowed in the Core League, mostly because the ships need to be exceptionally small and nimble, as all Core races travel across the galactic core, through an area of space that is itself twisted and warped by neutron stars and blackholes swirling towards the central supermassive blackhole. The Core League will require very complex wormhole configurations, and the race will mostly be about how the racers handle the shapes of the tunnel and its natural wobbling. Core League races should be the most visually interesting as you can see the blackholes and stars through walls of the wormholes.

Ship components:

Fuels:

        

  • Sugars, proteins and amino-acids: biochemical processes such as most crew need these
  • Long-chains: Petrochemicals that can be rapidly oxidized and explode. Primitive.
  • Methane: Easily located and scooped flammable gas that can be used to build longchain petrochemicals with applied chemistry and biology.
  • Fissile Matter: Uranium, Plutonium and Thorium (amongst others). These can produce power, but they are dangerous and most species require a lot of protection from the radiation they emit. Which tends to mean quite a lot of heavy shielding and safety systems.
  • Fusion Matter: Easy sourced from most larger gas giants brown dwarves or suns. And probably the most popular fuel in the galaxy.
  • Darkmatter: Found throughout space, but it congregates in halos and occasionally semi stable “lumps” unless disturbed. Dark matter can be used to generate power with a stable quantum singularity of the correct type. And represents the most advanced power sources in the galaxy, but a very tricky to build in small ships due to the immense weight of the dark matter particles themselves, and because dark matter does not respond like normal matter to a matter scoop and requires a dark matter scoop instead.
  • Zero-point energy: Essentially extracting free energy from quantum foam, theoretically it could have an almost limitless energy density, but in practice it will undoubtedly be quite restricted.
  • Protomatter: This uses a wormhole to reach back in time to a few moments after the big bang, and pulls forward some of the incredibly energy dense early universe. Small pulses of this power source can run entire planetary grids, and even nanosecond bursts can easily fry most power distribution grids, these are very powerful (once you can bootstrap start the wormhole extractor) but very dangerous to use right, and potentially can unmake the entire galaxy at any moment, or alter the fundamental structure of the universe. The few companies that own this technology do not share it except with each other, and are very careful about the specific coordinates the extractors use.

Power Plants:

  • Chemical: burns fuel with oxygen to generate electrical power
  • Fission power: uses fissile matter to generate electricity
  • Fusion power: fuses matter to generate either electrical power or plasma
  • Anti-matter annihilator: destroys matter to generate plasma
  • Mass- accelerator / Quantum singularity: consumes matter to generate plasma power.
  • Plasma converter, convert plasma into electrical power.

Ships can have electrical wiring and plasma conduits with different frequencies per metre.

Building wormholes…

Wormholes are constructed between two positions in the Galaxy

If the positions are the same the racecourse is elliptical with the a length based on the number of ships racing (more ships make bigger wormholes and orbitals that are small orbit the nearest moon, then planet then the sun then the barycentre of all suns, then finally the galactic centre).

The mass of the galaxy pulls and pushes the wormhole shapes, to mimic this the galaxy should be built with a multi-dimensional dimensional perlin noise. A cross section of the perlin noise can modify the perfect straight path of the wormhole. This is the first initial modification. This modification is multiplied by sine across its whole length to make variation 0 at the openings and much greater at the middle.

Next the frequencies of the ships are taken into account. Each ship has a scale that it is tuned to. The entire scale is used to calculate a number of different octaves of frequencies that each ship uses. When the ship selects a specific key in the scale that frequency is present in all octaves. All the frequencies apply to the entire tunnel (moving the tunnel as a lissajous perpendicular to the direction of travel for each octave again multiplied by a sine across the entire length to minimise wobbles at the ends and maximise them in the middle). These same frequencies also “wobble” the surface of the wall where the shield is in contact with the wall, with the wall in each section having a resonant frequencies based on the length, width and circumference of the wormhole. Ships are expected to have to change frequencies often to get better traction from the wall (and acceleration) but I’m hoping this will emerge naturally if the physics engine considers frictional forces as a harmonic. Musical Harmonies should probably be used, and I think will give bonuses based on whether the harmony chosen is a perfect key match (unison, perfect fifths and fourths and so on down to the most dissonant notes like a minor 12th or whatever the correct musical terms are). These ideally should again be emergent from the mathematics occurring, integer ratios can be compared Most Harmonic (Consonant): Ratios with very small integers, such as 1:2 (octave) or 2:3 (perfect fifth), are highly consonant.

Less Harmonic (Dissonant): Ratios involving larger integers, such as 7:5 (tritone), are considered more dissonant.

Modes in 12-Tone Equal Temperament

In the modern 12-tone equal temperament (12-TET) system, which is used for instruments like pianos and guitars, all intervals are standardized so that the octave is divided into 12 equal semitones. The frequency ratio for each semitone is the twelfth root of two (

212the 12th root of 2 end-root

212√).

In this system, the interval ratios for every mode relative to its tonic follow the same mathematical pattern:

Frequency Ratio=2(n/12)Frequency Ratio equals 2 raised to the open paren n / 12 close paren power

Frequency Ratio=2(𝑛/12)

where 'n' is the number of semitones above the tonic.

The seven standard modes use the following number of semitones to construct their scales:

  • Ionian: 0, 2, 4, 5, 7, 9, 11, 12 semitones
  • Dorian: 0, 2, 3, 5, 7, 9, 10, 12 semitones
  • Phrygian: 0, 1, 3, 5, 7, 8, 10, 12 semitones
  • Lydian: 0, 2, 4, 6, 7, 9, 11, 12 semitones
  • Mixolydian: 0, 2, 4, 5, 7, 9, 10, 12 semitones
  • Aeolian: 0, 2, 3, 5, 7, 8, 10, 12 semitones
  • Locrian: 0, 1, 3, 5, 6, 8, 10, 12 semitones

For example, the major third in any 12-TET major mode has a ratio of

2(4/12)2 raised to the open paren 4 / 12 close paren power

2(4/12) or

2(1/3)2 raised to the open paren 1 / 3 close paren power

2(1/3), which is approximately 1.2599:1, very close to the just intonation's 5:4 (1.25:1).

So how do we define a scale? From Chords and Tones I guess. So each planet needs 3 tones. Note 1 distance from the centre of the galaxy to edge, note 2 distance around the circle of the galaxy, note 3 distance above or below the plane of the ecliptic.

Technically the distance from the centre (radius) is actually revealing a wavelength (the length of that orbit around the galactic centre. This should also apply around the galaxy vertically, considering the orbit as a torus that defined the path around the galactic core. The same holds true for the previous of planets around the star, each one applies a wobble to the star, and those wobbles will be felt in the paths around those planets and stars and between them too.

Therefore each planet, or star, provides a chord (the first and last chord of the race). Perlin noise should provide the changes in between, although we need to know the most common changes too… by scale they look like this:

How does the wormhole technology work?

Who knows, but here’s how we are going to say the system works. A Vortex generator is a gravitic waveguide that allows a combination of at least two spinning vortex generators to create a wormhole. At least 2 small ships can orbit the wormhole entrance, they must essentially link their engines together (or on the cheap use a harpoon and hope) to coordinate orbiting the wormhole. The Vortex generators uses a gravitic waveguide material to “create” Dark Matter at centre point of the orbiting ships, and the navigation computers have to then calculate a phase angle and distance the wormhole is intended to project and reach toward another location, failure to connect will cause an orbital to be created, where the wormhole circles back (so smaller ships have much less chance of opening a wormhole out of a system). The targeting and spinning takes a proportional amount of time depending upon the distance and power available to the ships. Each ship system has a fundamental harmonic (and all crew including the player are also part of that harmonic systems, giving a key in a specific “octave”). At the moment that the wormhole creates an event horizon, the Vortex manipulators release a Dark Energy beam that forms the central structure of the vortex, holding the wormhole open from start to destination, at least you hope it does, in actual fact the Dark Energy provided will decay over time and eventually the wormhole will collapse, which means early races are going to be like 3 laps around a moon at most, 1 lap around a planet maybe, and 1 lap around a solar system if enough ships are working together.

This creates a visual start to the race, the ships form up into a circle, they may attach harpoons to a central point, and begin spinning, the navigators, tweak the angle of the orbit and align the systems, tuning the vortex generators to create the correct gate “here” and “there” (which might also be here). Then gravity lensing begins to affect the view, the ship immediately opposite seems to get bigger and look distorted, until a flash of light begins to appear at the centre, beams shoot out from the ships, and the funnel of a wormhole throat forms around the spinning ships, sucking them in via the harpoons (this would be the ideal time to disengage the harpoon) and the ships are thrown into the wormhole “tunnel” pushed away from the centre by the Dark Energy core (which will vaporize ship components as it expands matter if the two intersect, so it is a good job it pushes away the ships and the walls).

The Dark Energy core forms the centre line of the tunnel, and this is essentially a curve from start to end (a circle if they are the same place). This is essentially a conic cross-section (hyperbolic or elliptical typically). But then we also add a displacement field that is multi-dimensional perlin noise (we use rotational sections through the multidimensional perlin so that we have a repeating “random noise” - this creates a “lightning” like effect along the entire length (sine wave multiplier for the length to make the middle more mobile than the ends). Additionally the ships octaves also cause each point to have a number of lissajous that are applied to the whole length of the tunnel, but are strongest at the middle and less strong at the edges (same Sine multiplier).

Then the individual wall sections that surround the central Dark Energy Core also use the same Perlin noise (although probably a different 3 spatial dimensions from the multidimensional space) that is also looped to create a repeating pattern that is applied around the walls of the tunnel locally.

In game terms the walls are made from individual triangles where their vertices are all “wobbling” independently, but are affected by the central core’s completely different wobble. To make sure we don’t end up with the walls pushing back through themselves we actually only allow 2 dimensional movement or we limit the walls with some simple rules, so the perlin noise and lissajous push and pull the structure, but it physically can only move so far to respond to those changes. That will require a lot of potential development.
That’s even before we get to systems like, where the ships are in relation to the wall, should have an effect on the wormhole, effectively weighing the section they are on “down”, and causing some other point further away to flex in the opposite direction. That will require breaking the tunnel into nodes or similar to allow this. I guess it will depend on the octave and actual frequencies, with real wave physics being the ideal (if we can model the line as a guitar string essentially as one of the base models that we are using under the hood).

The wavelength of the wormhole has a curve that must be a multiple of the frequencies of the ships making the wormhole. That defines how much curve the conic has to have to define the path which then governs the natural harmonics across the entire wormhole, (that should also create the “song” that the wormhole “plays” that also has an impact on the places the wormhole samples for perlin noise).

/*

 * Continuous, Non-Linear Frequency-to-Color Converter (Sigmoid Smoothing)

 * Input: f_sound (Hz) - The frequency of the musical tone.

 * Output: HTML Hex Code String (e.g., "#FF0000").

 */

const curvedFrequencyToHex = (f_sound) => {

    // --- 1. Constants and Shift Calculation ---

    const C = 299792458; // Speed of Light in m/s

    const f_ref_sound = 440; // Hz (Reference A4)

    const f_ref_light_THz = 385; // THz (Reference A)

    const shift_factor = (f_ref_light_THz * 1e12) / f_ref_sound;

    const f_light_Hz = f_sound * shift_factor;

    // Wavelength (lambda) in nanometers (nm)

    const lambda = (C / f_light_Hz) * 1e9;

   

    // --- 2. Custom Boundary Conditions (IR/UV Perception) ---

    if (lambda > 780) {

        return "#000000"; // Black (Near-IR)

    } else if (lambda < 380) {

        return "#FFFFFF"; // White (Near-UV)

    }

   

    // --- 3. Non-Linear Sigmoid-Based RGB Calculation ---

   

    // Normalize Wavelength: Map 380nm to 780nm onto a 0 to 1 range.

    const lambda_min = 380;

    const lambda_max = 780;

    const normalized_lambda = (lambda - lambda_min) / (lambda_max - lambda_min);

   

    // Define the centers and widths for the three overlapping color curves.

    // These values are tuned to create a spectrum-like effect.

    // Center: Where the color is strongest (0.0=Violet, 1.0=Red)

    // Width: How wide the color's influence is.

    const R_center = 0.82;

    const G_center = 0.50;

    const B_center = 0.18;

    const width = 0.15; // Controls the steepness of the curves.

    // Sigmoid function (1 / (1 + e^-x)) is used to create a smooth S-curve transition.

    // The curve below is a Gaussian/Bell-shaped approximation that is smooth and continuous.

    const color_curve = (center) => {

        const exponent = -0.5 * Math.pow((normalized_lambda - center) / width, 2);

        return Math.exp(exponent);

    };

    // Calculate the raw component values

    let R_raw = color_curve(R_center);

    let G_raw = color_curve(G_center);

    let B_raw = color_curve(B_center);

   

    // Simple scaling/normalization to brighten the output:

    const scale = 255 / Math.max(R_raw, G_raw, B_raw);

   

    // --- 4. Final RGB and Hex Conversion ---

    const r = Math.round(Math.min(255, R_raw * scale));

    const g = Math.round(Math.min(255, G_raw * scale));

    const b = Math.round(Math.min(255, B_raw * scale));

    const componentToHex = (c) => {

        const hex = c.toString(16);

        return hex.length === 1 ? "0" + hex : hex;

    };

   

    return "#" + componentToHex(r) + componentToHex(g) + componentToHex(b);

};
